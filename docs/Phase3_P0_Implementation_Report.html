1. Executive Summary

Phase 3 P0 addresses the behavioral failures identified in the June 3 Telegram test session. The root cause was not lead detection — it was missing conversation state, turn routing, and response validation. P0 adds a session brain layer between Telegram ingress and the LLM, enforcing rules the prompt alone could not guarantee.

2. P0 Modules Delivered

#ModuleFileFunction
1Session State Managersession_state_manager.pyTracks user_id, company_id, language, name, phone, location, package_interest, lead_stage, pitch_count, phone_collected, escalation_requested per tenant session
2Turn Routerturn_router.pyRoutes: greeting, company_info, support, sales, objection, escalation, correction, memory_query, general_knowledge. Returns direct answers without LLM where safe
3Response Validatorresponse_validator.pyBlocks repeated package pitch, repeated phone request, repeated closing sentence, duplicate responses (>70% overlap), hallucinated phone numbers
4Language Locklanguage_lock.pyDetects Nepali vs English from user text; injects lock prompt so LLM does not randomly switch language
5Memory Validatormemory_validator.pySaves name only from high-confidence patterns; rejects "I am Indian I am Nepali"; phone saved only if present in source message

4. Pipeline Architecture (After P0)

Telegram update
  → main.py (update_id dedup — prevents double replies)
  → sanitize_user_text
  → detect_language → session state sync
  → extract_memory_facts → memory_validator → save memory
  → turn_router
      ├─ direct_answer? → response_validator → reply (no LLM)
      └─ memory_query?  → direct memory answer
  → lead extraction + upsert → session sync
  → sales mode (turn-aware, not always ON)
  → compose_system_prompt (+ language lock + session rules + turn hint)
  → OpenAI
  → response_validator (strip/blocks repeat pitch, phone, closing)
  → record session state + save chat turn
  → Telegram reply
